# Architecture Decisions: Stages 1-2

This document records the initial architecture decisions for the Stage 1 and Stage 2 lightweight assistant backend.

## 1. Backend Framework

**Decision:** FastAPI

**Reason:**

- Simple API development.
- Built-in OpenAPI docs.
- Strong Pydantic integration.
- Good fit for Cloud Run container deployment later.

## 2. Schema Validation

**Decision:** Pydantic v2

**Reason:**

- Explicit request and response models.
- Easy validation.
- Strong typing.
- Useful for a stable API contract.

## 3. Assistant Stub Response Behavior

**Decision:** The assistant endpoint returns a deterministic stubbed response that follows the final response schema. In Stage 2, search-like queries may return catalog-based mock results from the local synthetic catalog.

**Reason:** This allows API contract validation before implementing real retrieval, ranking, or AI logic.

## 4. Intent Detection

**Decision:**

- Not implemented in Stage 1.
- Future implementation may use a rule-based detector and optional LLM/Vertex AI provider.

**Reason:** This keeps Stage 1 focused and prevents premature complexity.

## 5. Product Retrieval

**Decision:**

- Real retrieval is not implemented in Stage 2.
- Future implementation may use hybrid keyword + vector retrieval.

**Reason:** Stage 2 prepares validated catalog data, but retrieval scoring and embeddings belong to later stages.

## 6. Cloud Deployment

**Decision:**

- Not implemented in Stage 1.
- Future target is Cloud Run, with optional Vertex AI and BigQuery Vector Search.

**Reason:** Stage 1 should define the contract first, then deployment will follow.

## 7. Synthetic Catalog Format

**Decision:** Stage 2 stores the synthetic product catalog as checked-in JSON at `app/data/products.json`.

**Reason:** A local JSON catalog is easy to inspect, deterministic in tests, and sufficient before introducing managed storage or vector infrastructure.

## 8. Catalog Validation

**Decision:** Catalog records are loaded through Pydantic v2 models before use.

**Reason:** Future retrieval code should be able to rely on stable product ids, partner coverage, positive prices, normalized tags, and rejected unknown fields.

## 9. Catalog Generation

**Decision:** The catalog is generated by `app/data/generate_synthetic_catalog.py` from structured product templates and a fixed random seed.

**Reason:** Deterministic generation keeps the checked-in catalog reproducible and avoids test churn from random data.

## 10. Filtering Before Retrieval

**Decision:** Stage 2 includes simple filtering utilities only.

**Reason:** Partner, category, price, tag, availability, and promotion filters are useful for Stage 3 retrieval preparation, while embeddings, vector search, and ranking belong to later stages.
